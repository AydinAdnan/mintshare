# TURN Setup

This project can pair devices through the signaling server without TURN, but reliable file transfer across mobile devices, different networks, hotspot connections, guest Wi-Fi, and restrictive NATs usually requires a TURN relay.

Without TURN, you may see these symptoms:

- sender and receiver appear connected, but the transfer never starts
- transfer works on one Wi-Fi but fails on mobile devices
- speed is unstable or extremely low
- desktop-to-desktop works, but mobile-to-desktop or mobile-to-mobile does not

## What TURN does

WebRTC first tries to establish a direct peer-to-peer path.

If the devices cannot reach each other directly, a TURN server relays the encrypted traffic between them.

For this app:

- signaling server: coordinates pairing and SDP/ICE exchange
- STUN server: helps discover network addresses
- TURN server: relays traffic when direct connection fails

## Environment variables used by this app

The frontend reads these values:

```dotenv
VITE_TURN_URLS=
VITE_TURN_USERNAME=
VITE_TURN_CREDENTIAL=
```

Example:

```dotenv
VITE_TURN_URLS=turn:global.relay.metered.ca:80?transport=udp,turn:global.relay.metered.ca:80?transport=tcp,turns:global.relay.metered.ca:443?transport=tcp
VITE_TURN_USERNAME=your-username
VITE_TURN_CREDENTIAL=your-password
```

Set these in:

- local `.env` for local testing
- Vercel environment variables for production frontend

## Recommended easiest option: Metered.ca

Metered is the fastest managed TURN option for this app.

### Steps

1. Create an account at Metered.
2. Create a new TURN application.
3. Copy the provided TURN URLs, username, and credential.
4. Set them as environment variables in Vercel.

Typical values look like this:

```dotenv
VITE_TURN_URLS=turn:global.relay.metered.ca:80?transport=tcp,turn:global.relay.metered.ca:80?transport=udp,turns:global.relay.metered.ca:443?transport=tcp
VITE_TURN_USERNAME=example-user
VITE_TURN_CREDENTIAL=example-password
```

### Vercel setup

In your Vercel frontend project, add:

```dotenv
VITE_SIGNAL_URL=wss://fileshare-signal.onrender.com
VITE_TURN_URLS=turn:global.relay.metered.ca:80?transport=tcp,turn:global.relay.metered.ca:80?transport=udp,turns:global.relay.metered.ca:443?transport=tcp
VITE_TURN_USERNAME=example-user
VITE_TURN_CREDENTIAL=example-password
```

Then redeploy the frontend.

## Self-hosted option: coturn on a VPS

If you want full control, run coturn on a VPS.

### Requirements

1. A VPS with a public IP
2. A domain or subdomain such as `turn.your-domain.com`
3. Open ports:
   - `3478` TCP/UDP
   - `5349` TCP for TLS
   - relay UDP range, commonly `49152-65535`

### Install coturn on Ubuntu

```bash
sudo apt update
sudo apt install coturn -y
```

Enable coturn:

```bash
sudo sed -i 's/#TURNSERVER_ENABLED=1/TURNSERVER_ENABLED=1/' /etc/default/coturn
```

### Basic coturn config

Edit `/etc/turnserver.conf` and configure:

```conf
listening-port=3478
tls-listening-port=5349

fingerprint
use-auth-secret
static-auth-secret=replace-with-a-long-random-secret

realm=turn.your-domain.com
server-name=turn.your-domain.com

total-quota=100
bps-capacity=0
stale-nonce=600

cert=/etc/letsencrypt/live/turn.your-domain.com/fullchain.pem
pkey=/etc/letsencrypt/live/turn.your-domain.com/privkey.pem

no-multicast-peers
no-cli
```

Restart coturn:

```bash
sudo systemctl restart coturn
sudo systemctl enable coturn
```

### Authentication note

Coturn is commonly used with time-limited credentials generated by your backend.

This app currently expects a static username and credential pair in the frontend environment, so a managed TURN provider is simpler unless you also want to extend the signaling backend to mint temporary TURN credentials.

## Testing TURN

After configuring TURN:

1. Redeploy the frontend.
2. Open sender on one network and receiver on another network if possible.
3. Start a transfer from desktop to mobile.
4. If it still fails, verify your TURN provider values and make sure the frontend was redeployed after the env changes.

## Common mistakes

1. Using `https://` instead of `wss://` for `VITE_SIGNAL_URL`
2. Leaving TURN values blank and expecting mobile cross-network transfers to work reliably
3. Setting TURN values in local `.env` but not in Vercel
4. Updating env vars but forgetting to redeploy the frontend
5. Using only one TURN URL instead of providing both TCP and TLS-capable endpoints

## Recommended production setup for this project

Use this split:

1. Frontend on Vercel
2. Signaling server on Render
3. Managed TURN provider such as Metered

That gives you:

1. simple frontend hosting
2. persistent WebSocket signaling
3. much better mobile and cross-network reliability
